// 02 — decisión de tecnologías

Por qué elegimos este stack

Cada tecnología fue elegida por razones técnicas concretas, no por popularidad.

🍃

MongoDB Atlas

Base de datos NoSQL orientada a documentos. Los documentos JSON encajan naturalmente con el modelo de tareas, que tiene campos opcionales y estructura variable.

¿Por qué no PostgreSQL? Nuestros datos no tienen relaciones complejas entre tablas; un documento por tarea es suficiente. MongoDB Atlas ofrece 512 MB gratuitos y despliegue en segundos.
🚀

Render

Plataforma PaaS que soporta Node.js, Python, Go y más. Despliega automáticamente desde GitHub en cada push.

¿Por qué no PythonAnywhere? Render es agnóstico al lenguaje, tiene HTTPS automático, variables de entorno en el panel y URL pública instantánea.
⚡

Express + Mongoose

Framework minimalista de Node.js para la API REST. Mongoose añade validación de esquemas y una capa ODM sobre MongoDB.

¿Por qué no Flask/Django? El equipo tiene más experiencia en JavaScript. Un solo lenguaje (JS) en todo el stack reduce la fricción de contexto.
📄

GitHub Pages

Hosting gratuito para la versión estática. Sirve HTML/CSS/JS directamente desde el repositorio, sin configuración de servidor.

Rol en la arquitectura: Muestra esta página de documentación y puede servir como shell del frontend que consume la API de Render vía fetch().

// 03 — comparativa

GitHub Pages vs Render

Ambas plataformas son gratuitas pero sirven propósitos distintos. Usamos las dos en combinación.

Característica GitHub Pages Render (free tier)
Tipo de contenido Solo estático (HTML/CSS/JS) Estático y dinámico (backend completo)
Backend (Node, Python…) ✗ No soportado ✓ Sí
Conexión a BD ✗ Imposible directamente ✓ MongoDB, PostgreSQL, Redis
Variables de entorno ✗ No ✓ Panel de control
Velocidad de despliegue ~segundos ~minutos (cold start)
Disponibilidad 24/7 ✓ Sin restricciones Se duerme por inactividad
HTTPS automático ✓ Sí ✓ Sí
Caso de uso ideal Portafolios, docs, landing pages APIs REST, apps full-stack con BD nuestro backend

Conclusión: GitHub Pages es perfecto para contenido estático y documentación (esta página), mientras que Render maneja el backend con MongoDB. La combinación de ambos da una arquitectura completa sin costo.

// 04 — implementación

Código principal

Los fragmentos clave del servidor Express con MongoDB.

Conexión a MongoDB Atlas y modelo de tarea:

// server.js
const express = require('express');
const mongoose = require('mongoose');
require('dotenv').config();

// Conexión a MongoDB Atlas (URI en variable de entorno)
mongoose.connect(process.env.MONGODB_URI)
  .then(() => console.log('✅ MongoDB conectado'));

// Esquema de tarea
const TaskSchema = new mongoose.Schema({
  title:     { type: String, required: true },
  done:      { type: Boolean, default: false },
  createdAt: { type: Date,   default: Date.now }
});
const Task = mongoose.model('Task', TaskSchema);

Endpoints CRUD:

const router = express.Router();

// GET todas las tareas
router.get('/', async (req, res) => {
  const tasks = await Task.find().sort({ createdAt: -1 });
  res.json(tasks);
});

// POST nueva tarea
router.post('/', async (req, res) => {
  const task = await Task.create({ title: req.body.title });
  res.status(201).json(task);
});

// PATCH marcar como completada
router.patch('/:id', async (req, res) => {
  const task = await Task.findByIdAndUpdate(
    req.params.id,
    { done: req.body.done },
    { new: true }
  );
  res.json(task);
});

// DELETE eliminar tarea
router.delete('/:id', async (req, res) => {
  await Task.findByIdAndDelete(req.params.id);
  res.json({ ok: true });
});
